# Phase 8B Step 4: Material-Universe Failure Analysis

## Scope

This analysis covers the current Material-Universe generation and gate path
after Facts Gate acceptance. It is reactor-neutral and is based on the current
implementation in `openmc_agent/plan_builder`, `openmc_agent/plan_investigation`,
and the existing closed-loop tests. No code changes are included in this
analysis step.

## Current Pipeline

The current controlled path is:

1. Accepted Facts are required before the Material-Universe Gate can run.
2. Materials and Universes patches are generated by the existing patch
   generation pipeline.
3. `build_material_universe_binding_view()` parses the three valid patches and
   constructs material records, universe records, cell bindings, and fuel
   variant bindings.
4. `run_material_universe_preflight()` runs schema validation, local material
   checks, cross-patch binding checks, fuel-variant checks, and radial checks.
5. `build_material_universe_evidence_pack()` builds the contract matrix and
   evidence items.
6. `run_material_universe_review()` calls one combined
   `MaterialUniverseReviewModelOutput` reviewer.
7. Deterministic findings and reviewer findings are merged. Blocking findings
   are routed through the Phase-3B retry controller.

The repository already contains two useful upstream requirement paths:

- `MaterialGenerationRequirementSet` from `GeometryComponentInventory`.
- `InventoryUniverseRequirementSet` from inventory radial profiles.

The existing `MaterialBindingSkeleton` and `UniverseBindingSkeleton` compile
binding slots from those requirement sets, but they are binding-oriented
objects. They do not yet provide the requested unified requirement skeleton
with explicit provenance, source-derived immutable identity, and the five
requested resolution states.

## 1. Fields Still Dependent on LLM Guessing

The following values are still exposed to patch generation or semantic review
when the deterministic inputs are not propagated into a typed requirement
object:

- Material composition detail and isotope-level composition.
- Composition basis for compounds and mixtures.
- Density value when Facts/Evidence do not provide a resolved density.
- Material-to-requirement association when a generated material omits or
  changes `source_variant_id`.
- Universe identity and geometry-profile association when generated Universes
  do not carry inventory profile metadata.
- Required cell role and material-role bindings when a Universe is generated
  from a compact prompt rather than a requirement skeleton.
- Localized insert Universe association when the generated Universe does not
  preserve the source insert requirement or protected path.
- Whether multiple fuel variants remain distinct in Universes when variant
  identity is not copied into every generated fuel Universe.

The LLM may fill composition details, missing source-backed information, and
explicit assumptions. It must not select the existence, count, role, or source
variant identity of a required material or Universe.

## 2. Fields Deterministically Derivable from Evidence, Inventory, and Facts

The following values can be compiled without an LLM:

### Materials

- Required material count, after deterministic deduplication by role,
  source-variant identity, and localized-insert requirement.
- `material_id` as a deterministic requirement identifier or a protected
  existing binding.
- Material role.
- Fuel `source_variant_id`.
- Localized-insert material role and source requirement.
- Required density presence, density provenance requirement, and whether a
  density is unresolved.
- Required isotope set when accepted Facts or Evidence explicitly declares it.
- Geometry usage, including the profiles, components, and cells that consume
  the material.
- Source claim IDs, source span IDs, inventory hash, and accepted Facts hash.

### Universes

- Universe requirement count from resolved inventory radial profiles.
- `universe_id` as a deterministic requirement identifier or protected
  existing binding.
- Component kind, profile kind, and geometry profile.
- Required materials and required material roles.
- Required cell roles and radial layer roles.
- Fuel variant identity for variant-specific profiles.
- Localized insert requirement IDs and protected through-path roles.
- Required source requirement IDs and evidence references.
- Whether the requirement is unresolved because the source profile is
  unresolved.

The current inventory and requirement compilers already expose most of these
inputs. The missing step is to make the generation contract consume typed
requirement skeletons rather than allowing the patch generator to infer them
from prose or compact summaries.

## 3. Deterministic Checks

The following checks belong in Python and must fail closed before reviewer
semantic judgment:

- Required material missing.
- Required material count or duplicate source variant mismatch.
- Required material role mismatch.
- Required density missing, invalid, or lacking required provenance.
- Required isotope missing or unresolved for a declared compound/mixture.
- Fuel variant material missing, duplicated, or assigned to another variant.
- Required Universe missing for a resolved geometry profile.
- Duplicate Universe identity or duplicate profile binding.
- Universe material binding missing or referencing an unknown material.
- Required cell role or required material role missing.
- Fuel variant collapse across active fuel Universes.
- Localized insert Universe orphaned from its insert requirement.
- Guide/instrument tube binding missing when the inventory requires the host
  component and through-path role.
- Protected source paths or source requirement IDs missing from generated
  Universe metadata.
- Universe profile, component kind, or source requirement mismatch.
- Inventory hash, Facts hash, or requirement-set hash drift.
- Partial fragment or missing manifest coverage.

The current `material_universe_preflight.py` already detects several of these:
duplicate IDs, invalid density, unknown material references, radial overlap,
fuel-variant absence/collapse indicators, role mismatch, and some structural
coverage. It does not yet cover all typed requirement fields or all protected
path and localized-insert invariants.

## 4. Main Sources of Current MU Failure

The current failure is downstream of Facts acceptance. The latest real
Facts-only canary records `facts=accepted` and then fails in subsequent patch
generation before a complete MU Gate artifact is produced. This means the
current run does not demonstrate a MU reviewer rejection; it demonstrates that
the Materials/Universes generation boundary is not yet robust enough to produce
the inputs required by the MU Gate.

The implementation-level failure risks are:

1. **Generation starts from patch output rather than a complete typed
   requirement contract.** Existing requirement sets are available, but the
   binding skeleton is not yet the authoritative generation input.
2. **Material and Universe identity can drift.** Existing skeletons protect
   fields after a slot is compiled, but generated candidates still need a
   deterministic requirement-to-output reconciliation step.
3. **The MU reviewer is monolithic.** One prompt reviews material records,
   universe records, bindings, variants, and contract rows. This increases
   output size and makes coverage and schema failures harder to localize.
4. **Some deterministic failures are mixed with semantic findings.** The
   current executor converts all preflight issues into generic
   `PlanReviewFinding` objects, often with both Materials and Universes as
   affected patch types. This weakens targeted retry ownership.
5. **Requirement coverage is not uniformly enforced.** Inventory preflight
   checks profile and role coverage, while the MU binding view derives a
   separate Facts-based contract list. The two views need one canonical
   requirement skeleton and one coverage fingerprint.
6. **Retry can identify broad owners, but not every required sub-owner.** The
   canonical issue policy maps most codes to `materials` or `universes`, but
   missing density, isotope resolution, source-variant mismatch, localized
   insert orphan, and protected-path failures need stable issue codes and
   targeted request metadata.

## 5. Retry Owner Assessment

The retry controller is structurally capable of targeted owner retries:

- `material_universe_issue_policy.py` is the canonical MU code-to-owner map.
- `retry_owner_policy.py` adapts those owners to executable retry policies.
- `normalize_retry_request()` ignores LLM-selected owners and uses the Python
  registry.
- The dependency graph invalidates downstream patches and gates after an owner
  patch changes.
- Facts, Placement, and Axial are not regenerated for a Materials-only or
  Universes-only MU retry unless dependency invalidation explicitly requires a
  downstream rebuild.

However, the current integration has two gaps:

- The executor often reports deterministic MU issues with broad affected patch
  types instead of attaching the canonical owner and requirement target.
- The retry owner registry does not yet contain all requested new issue codes,
  especially density missing, compound isotope unresolved, localized insert
  Universe missing/orphan, and protected path missing.

Unknown issue codes fail closed, which is correct for safety but currently
turns otherwise repairable MU failures into blocked planning failures. The
stabilization work should add the missing deterministic codes and route
metadata, not weaken the fail-closed behavior.

## Recommended Phase 8B Step 4 Design

1. Add `MaterialRequirementSkeleton` and `UniverseRequirementSkeleton` as
   immutable, source-derived typed models. Keep the existing binding skeletons
   as compatibility-layer binding outputs, or make them consume the new
   requirement skeletons without changing Facts behavior.
2. Compile the skeletons only after Facts is accepted and from the shared
   Evidence/Inventory/Facts inputs.
3. Add a deterministic requirement-to-patch preflight that runs before any
   reviewer and produces owner-specific issue codes and affected paths.
4. Split the reviewer into Material, Universe, and Binding calls with separate
   schemas, fingerprints, coverage lists, and artifacts. The reviewers must not
   repeat deterministic checks or choose owners/actions.
5. Extend retry owner policy with the new issue codes and preserve the rule
   that MU retries never regenerate Facts, Placement, or Axial directly.
6. Run focused unit tests first, then the required full non-OpenMC suite,
   compile check, fake benchmark, and three controlled real MU canaries.
