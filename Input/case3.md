# 算例3：C5G7-3D 四分之一堆芯 OpenMC 模型（LLM 友好版）

请根据下面的建模约束，生成一个能够完整定义 C5G7-3D 四分之一堆芯的 OpenMC 模型，并同时给出必要的几何一致性校验和低计算成本运行诊断方案。

本输入文件仍是 Markdown 文档，不提供可直接照抄的 Python 脚本、YAML、JSON 或其它结构化中间表示；但是为了降低 LLM 在 17×17 棒位图展开时的转录错误，本文档对棒位布局使用 **符号表、坐标约定、canonical pin map 和硬性计数表**。生成模型时应由智能体把这些 Markdown 约束解析为材料、曲面、栅格、单元、宇宙、几何、设置和计数校验。

本算例用于测试 openmc-agent 是否能够从自然语言与半结构化 Markdown 约束中恢复一个较大规模三维反应堆基准模型。目标不是进行高统计量物理基准验证，而是验证自动建模流程能否生成完整几何、导出 OpenMC XML、通过几何检查，并用低计算成本本征值运行排查明显建模错误。

## 0. LLM 解析总规则（必须优先遵守）

本节是为了避免 LLM 在展开 17×17 组件棒位图时发生镜像、上下翻转、下半部分误填、计数正确但空间位置错误等问题。

### 0.1 坐标约定

所有组件级棒位图都使用同一坐标约定：

- 行号为 `R01` 到 `R17`，从工程图 **上方到下方**。
- 列号为 `C01` 到 `C17`，从工程图 **左侧到右侧**。
- `R01C01` 是工程图左上角。
- 如果生成 schema 中的 `overrides` 坐标，应使用 0-index：
  - `row = R - 1`
  - `col = C - 1`
  - `row 0` 是最上行；
  - `col 0` 是最左列。
- 不得把 OpenMC RectLattice 的内部 y 方向顺序与本文档工程图行序混淆。若使用 OpenMC `RectLattice.universes`，必须显式说明是否需要反转行序以保持工程图空间位置正确。

### 0.2 大型矩阵的推荐 IR 表示

对于 17×17 组件棒位图，**不要优先手写完整 17×17 `universe_pattern`**。推荐使用紧凑表达：

- `shape = (17, 17)`；
- `fill_universe = <默认 pin universe>`；
- `overrides = {universe_id: [(row, col), ...]}`；
- `expected_counts = {...}`。

只有当系统必须输出完整矩阵时，才根据本文档的 canonical pin map 确定性展开。展开时不得依赖“看起来对称”来自行补全，必须逐行读取 `R01` 至 `R17`。

### 0.3 棒位图优先级

若同一组件同时出现自然语言描述和 canonical pin map：

1. **canonical pin map 优先级最高**；
2. 硬性计数表优先级第二；
3. 自然语言解释只作为辅助说明；
4. 如果自然语言与 canonical pin map 冲突，以 canonical pin map 为准，并在输出说明中指出采用了 canonical pin map。

### 0.4 必须进行的棒位校验

生成模型后必须至少检查：

- UO2 组件：`U = 264, G = 24, F = 1, total = 289`。
- MOX 组件：`A = 64, B = 100, C = 100, G = 24, F = 1, total = 289`。
- 四分之一堆芯活性区：两个 UO2 组件 + 两个 MOX 组件，总棒位 `1156`。
- 四分之一堆芯燃料棒总数：
  - UO2 = 528；
  - 4.3% MOX = 128；
  - 7.0% MOX = 200；
  - 8.7% MOX = 200；
  - 导向管 = 96；
  - 裂变室 = 4；
  - 燃料棒总数 = 1056。

如果任一计数或关键坐标不匹配，应视为模型生成失败，不能通过重命名材料、忽略棒位或改变物理布局来掩盖。

## 一、基准题背景和适用范围

本算例对应 OECD/NEA C5G7 MOX 基准题的三维四分之一堆芯形式。该基准题用于考察输运程序在不对燃料棒以上尺度进行空间均匀化的情况下处理 UO2 与 MOX 组件界面、17×17 棒阵列、径向反射层和三维轴向边界的能力。

本输入文件要求建立的是未插棒的三维四分之一堆芯版本。堆芯径向由 2×2 个燃料组件构成，外侧有一圈水反射层。每个燃料组件是 17×17 棒位阵列。四个燃料组件拼接后，活性燃料棒位阵列尺寸为 34×34。轴向方向包含燃料有效区和顶部水反射层，底部采用反射边界，上部外表面采用真空边界。

C5G7 原始基准题使用七群宏观截面和均匀化的棒内材料语义，其中燃料、包壳和间隙在圆柱内被作为等效混合材料处理，圆柱外是慢化剂水。蒙特卡罗输运的常规做法是连续能模拟，因此本算例不使用 C5G7 七群宏观截面，而是采用连续能核素材料卡：在保持 C5G7 几何与排布不变的前提下，把每个材料区域替换为连续能核素组成、密度、温度和热散射处理。

由于 C5G7 原始基准只给出七群均匀化截面、并未提供连续能核素组成，本算例给出的连续能材料数据是本算例为蒙卡可运行性确定的核素组成定义，不是 C5G7 基准标准数据；低成本运行结果只能用于程序诊断，不能当作高统计量 C5G7 物理基准验证。

## 二、总体建模目标

请建立一个 OpenMC 三维模型，使它能够完整表示 C5G7-3D 四分之一堆芯。模型至少应满足以下要求：

1. 生成 34×34 的活性棒位阵列，并保证它由四个 17×17 燃料组件拼接而成。
2. 区分 UO2 组件和 MOX 组件，并按照四分之一堆芯中 2×2 的组件排布放置。
3. 在燃料组件外侧设置一圈水反射层。水反射层厚度等于一个组件宽度，即 21.42 cm。
4. 设置燃料有效区和顶部水反射层两个轴向区域。燃料有效高度为 192.78 cm，顶部水反射层高度为 21.42 cm，总高度为 214.20 cm。
5. 把反射边界和真空边界清楚映射到 OpenMC 的曲面边界条件。对称面应为反射边界，外部泄漏面应为真空边界。
6. 通过 OpenMC XML 导出、几何检查和低计算成本本征值运行诊断，确认没有明显 XML 解析错误、几何重叠、未闭合区域、lost particle 或 ERROR 级错误。

| 单个棒位栅元边长 / pitch | 1.26 |
| 燃料棒、导向管、裂变室圆柱半径 | 0.54 |
| 单个组件棒位数 | 17×17 |
| 单个组件边长 | 21.42 |

| 四分之一堆芯径向总边长 | 64.26 |

| 总高度 | 214.20 |

轴向方向从底部到顶部分为两个连续区间：

1. `z = 0.00 cm` 到 `z = 192.78 cm`：燃料有效区。该区间内保留径向燃料组件和径向水反射层。
2. `z = 192.78 cm` 到 `z = 214.20 cm`：顶部水反射层。该区间内整个横截面都应填充为水。

顶部水反射层上方是真空边界。燃料有效区下方是反射边界。

## 四、径向组件排布

四分之一堆芯的活性燃料组件区包含四个 17×17 组件，排布为两个 UO2 组件和两个 MOX 组件。按工程图视角描述时：

- “上方”为北；
- “下方”为南；
- “左侧”为西；
- “右侧”为东。

### 4.1 活性 2×2 燃料组件区

| 位置 | 组件类型 |
|---|---|
| 左上 / 西北 | UO2 |
| 右上 / 东北 | MOX |
| 左下 / 西南 | MOX |
| 右下 / 东南 | UO2 |

也就是：

```text
活性 2×2 组件区：

[ UO2 ][ MOX ]
[ MOX ][ UO2 ]
```

### 4.2 3×3 组件宽度区域中的水反射层

在活性 2×2 组件区的外侧设置一圈水反射层。用 3×3 的组件宽度区域描述整个四分之一堆芯：

```text
[ UO2 ][ MOX ][ H2O reflector ]
[ MOX ][ UO2 ][ H2O reflector ]
[ H2O ][ H2O ][ H2O reflector ]
```

右侧一列为水反射层，下侧一行为水反射层，右下角也是水反射层。径向整体区域宽度为三个组件宽度，即 64.26 cm。

### 4.3 边界方向

本模型采用非负坐标约定时：

- 左侧边界是对称面，使用 reflective；
- 上侧边界是对称面，使用 reflective；
- 右侧外表面是泄漏面，使用 vacuum；
- 下侧外表面是泄漏面，使用 vacuum。

如果实现中采用中心坐标或其它坐标约定，也必须保持同一物理关系：两个相交的内侧对称面为反射边界，外侧水反射层的外表面为真空边界。不得把对称面和外部泄漏面弄反。

## 五、棒位类型和材料区域

模型中至少应包含七类棒位或区域：

| 符号/类型 | 物理含义 | 材料名 |
|---|---|---|
| U | UO2 燃料棒 | `uo2` |

| B | 7.0% MOX 燃料棒 | `mox7` |

几何上沿用 C5G7 简化：

- 每个燃料棒位分为两部分：
  - 半径 0.54 cm 以内为燃料材料；
  - 半径 0.54 cm 以外到 1.26 cm 方形栅元边界之间为水。

- `set_density("g/cm3", ...)`
- `add_nuclide(...)`

- `add_macroscopic`
- `density_unit="macro"`

### 7.5 MOX 关键坐标校验

1. `R01` 全部为 `A`。
2. `R17` 全部为 `A`。这是过去最容易漏掉的行，不能被填成 `B` 或其它符号。
3. `C01` 和 `C17` 两列全部为 `A`。
4. `R09C09` 必须为 `F`。
5. 导向管 `G` 的坐标必须与 UO2 组件完全相同：
   - R03: C06, C09, C12
   - R04: C04, C14
   - R06: C03, C06, C09, C12, C15
   - R09: C03, C06, C12, C15
   - R12: C03, C06, C09, C12, C15
   - R14: C04, C14

## 九、轴向分层和三维闭合

模型必须在轴向上闭合。燃料有效区从底部开始，延伸 192.78 cm。在该区间内，径向 2×2 燃料组件和外侧水反射层同时存在。顶部水反射层紧接燃料有效区，厚度 21.42 cm，从 192.78 cm 延伸到 214.20 cm。在顶部水反射层内，不应继续放置燃料棒、导向管或裂变室，整个横截面应为水。

底部边界为反射边界，顶部边界为真空边界。燃料区与顶部水反射层之间的内部轴向平面不应设置为真空或反射边界，它只是普通内部分界面。

生成几何时必须避免：

- 顶部水反射层漏建；
- 燃料棒圆柱无限延伸到顶部水反射层；
- 轴向内部平面被错误设置为 vacuum 或 reflective；
- 轴向 cell 未闭合。

## 十、边界条件

该四分之一堆芯模型有两个径向对称面和两个径向外表面。两个对称面应采用反射边界，两个外表面应采用真空边界。径向外表面位于水反射层外侧，而不是活性燃料组件区外侧。

边界条件表：

| 边界 | 物理含义 | 边界类型 |
|---|---|---|
| 左侧径向面 | 对称面 | reflective |
| 上侧径向面 | 对称面 | reflective |
| 右侧径向外表面 | 外部泄漏面 | vacuum |
| 下侧径向外表面 | 外部泄漏面 | vacuum |
| 底面 | 轴向反射面 | reflective |
| 顶面 | 轴向泄漏面 | vacuum |

内部组件边界、棒位边界、燃料区与顶部水反射层之间的分界面不应带有 vacuum 或 reflective 边界条件。

## 十一、OpenMC 建模组织要求

openmc-agent 可采用适合的 OpenMC Python API 组织方式，但生成结果必须满足以下语义。

### 11.1 pin universe

每类棒位用独立的 pin universe 或等效组织方式表示：

- `uo2_pin`
- `mox43_pin`
- `mox7_pin`
- `mox87_pin`
- `guide_tube_pin`
- `fiss_chamber_pin`
- `water_reflector` 或等效全水区域

这些 pin universe 必须能被单独识别和计数。

### 11.2 assembly lattice

每个 17×17 组件应由对应的棒位图生成：

- 两个 UO2 组件复用同一种 UO2 组件定义；
- 两个 MOX 组件复用同一种 MOX 组件定义；
- UO2 和 MOX 不得混用同一个 lattice pattern；
- MOX lattice 必须使用第七节 canonical pin map。

### 11.3 core-level layout

堆芯级组织应表达：

```text
[ UO2 ][ MOX ][ H2O reflector ]
[ MOX ][ UO2 ][ H2O reflector ]
[ H2O ][ H2O ][ H2O reflector ]
```

燃料有效区内使用该径向布局。顶部水反射层内整个横截面为水。

### 11.4 RectLattice 行序

使用 OpenMC RectLattice 时，应特别注意 OpenMC 的 lattice 行列顺序与工程图从上到下、从左到右描述之间的差异。生成器必须显式处理 y 方向顺序，不能把组件上下翻转后仍认为模型正确。

在生成说明中必须写明：

- 文档中的 `R01` 是否对应工程图最上行；
- schema 或 OpenMC lattice 中如何映射该行；
- 是否对 `universe_pattern` 做了行反转；
- 如何验证裂变室和导向管未发生上下翻转。

## 十三、常见歧义和修复要求

生成过程中出现 lattice 或 universe 填充歧义时，应优先通过显式坐标、显式边界和显式分区修复，而不是依赖 OpenMC 默认外延。

特别要处理以下问题：

1. **RectLattice 行顺序歧义**  
   OpenMC 的 RectLattice 行顺序与工程图行顺序不同，必须在生成时说明并修正索引映射。

2. **MOX pin map 转录错误**  
   如果 MOX 计数或空间位置错误，只能修复 MOX lattice pattern 或 compact overrides。不得修改材料、密度、边界条件、core layout 或运行设置来掩盖错误。

3. **水反射层覆盖范围**  
   堆芯级 3×3 区域中的水反射层可作为全水组件宽度块处理，或用显式水 cell 处理；无论哪种方法，都必须保证它覆盖活性组件外侧的一整圈水反射区域。

4. **轴向燃料区和顶部水层**  
   燃料区和顶部水反射层使用不同的轴向 cell 或 universe 区分。不能让 pin universe 中的圆柱区域无界延伸到顶部水反射层。

5. **边界条件误放置**  
   反射边界和真空边界的平面应只定义在整体模型外表面。内部组件边界、棒位边界和燃料区与顶部水反射层之间的分界面不应带有真空或反射边界条件。

6. **局部 universe 曲面作用域**  
   由于不同组件共享同一 pin surface 导致空间区域定义冲突时，应改为在局部 universe 内定义相对坐标 pin surface，或确保每个局部 universe 的曲面作用域符合 OpenMC 几何语义。
